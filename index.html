git 高级命令备忘录

16Oct

git 命令的一些备忘录

常用命令

  • 显示分支的url: git branch -v
  • 重命名分支名字: git branch -m realise
  • 删除远端分支: git push origin :branchname
  • 清理本地中，在服务器以前有过但现在不存在了的分支: git remote prune

存储 git stash

  • 显示列表: git stash list
  • 存储: git stash
  • 导出: git stash pop
  • 导出前面几个: git stash apply stash@{1}

文件系统中保存、查看

  • 存储起来：echo 'test' | git hash-object -w --stdin
  • 存入tag中，以备使用: git tag myfile HAS-1
  • 查看文件内容: git cat-file blob myfile

git log

  • 只查看改动的文件: git log --stat
  • 更好看的显示别名： git config --global alias.lol "log --pretty=oneline --abbrev-commit --graph --decorate"
  • 查找log: git log --grep='hello' -S='todo' --author="colin" --since="2.month.ago" --until="1.day.ago" --all-match

版本回退

  • 回退最后一次提交：git commit --amend
  • 类似reverge: git reset HEAD filepath
  • 多次回退：git checkout newbrach;git reset --hard HEAD~2
  • 修改分支的指向 git update-ref refs/heads/Foo SHA1
  • 如果你提交了一些内容到你的分支（也许你从SVN导入了一些旧仓库），然后你希望把某个文件从历史记录中全部删掉：git filter-branch --tree-filter 'rm -f *.class' HEAD
  • 如果你已经推送到origin了，但之后提交了一些垃圾改动，你也可以在推送前在本地系统里这样做：git filter-branch --tree-filter 'rm -f *.class' origin/master HEAD

查找文件修改者

  • 查看file的来源(file比如是合并来的): git blame -C file
  • gui 进行操作：git gui blame file

git 的维护

  • git垃圾回收: git gc
  • git 完整性检查 git fsck --full

从最短识别非现代浏览器语句引发的知识串烧

24Sep

序

今天偶尔在网上看到了这么一个语句var isIE =!-[1,],感觉很有意思，所以写了这一篇文章，记录下当时我的感想。

分析

  1. 首先这一个语句是有问题的，他并不能识别ie9+以上浏览器（我们有时候也会称之为现代浏览器），至于为什么不能识别，我们下文会有具体的分析，至于为什么会起这么个引人误会的isIE这个变量，是因为这个语句产生时的时间是还没有出现ie9呢（–！暗汗一下）。

  2. 这个浏览器识别标志是根据浏览器的特性来识别的，我们知道浏览器识别可以有两种办法（我所知道的，亲们有其他的方法，欢迎@我）。

    • 第一种办法是用window.navigator这个属性来进行正则匹配进行识别，可以通过它来识别浏览器的种别，同时可以识别浏览器的版本号

    • 第二种办法就是根据各浏览器的解析特性不同，而进行判别，这种方法可以很方便的分别浏览器的版本，但是会有几个问题出现。

      1. 可能会随着浏览器的升级而失效，增加后期的维护和升级成本。比如文中的这个语句，在ie8统治的时代会很cool，但是当ie9出现后，会随之出现各种让你伤心的bug的。

      2. 就是当需要不仅识别浏览器，还需要识别浏览器的版本时，你可能需要用到很多的条件，就没有第一种方法的便利了。

      3. 增加代码的阅读难度系数，当你写的代码给另外一个基础不是很扎实的时候，我想维护代码的人一定云里雾里的（纯发牢骚。。。）

从另一个角度谈js的面向对象编程（纯语法角度）

21Sep

前言

想写这篇这文章，已经很久很久了，起源是因为，从接触js开始，查看的网上，对于面向对象的解释都大同小异，都是从js语言特性的角度来分析的，什么作用域链，原型链巴拉巴拉的。。。

其实，这些文章很不错，但是，私下里总觉得少了什么样子似得，直到跟一个朋友唠嗑的时候，才意识到哪里出了问题。

场景大概是这样子的，一个写后端c系语言的朋友，来找我抱怨js的东东，说看了一遍书中对js面向编程的详解，觉得好郁闷，问我啥叫原型链啊，prototype到底干嘛用的。

我跟他解释了半天，什么作用域的构造了巴拉拉的一大顿，不得其解，然后跟他说，这个就像 class中的:: 运算符一样。所有定义在prototype上的，都会被对象继承，作为类外定义，朋友豁然开朗。

所以，私以为，把js中面向对象的一些东西，用真正后台语言的词汇来近似的解释一遍，感觉对从后端转前端的猿们来说，还是可以提高快速理解的吧，同时也能够验证自己是否真正从语法角度掌握了js的面向对象。
